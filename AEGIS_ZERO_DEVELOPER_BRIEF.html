This repo now contains a working trusted-kernel foundation for a local-first autonomous agent system: trusted retrieval, deterministic tool policy, approval gating, policy-enforced tool execution, orchestration, runtime config centralization, and harness-level regression coverage.

Architecture

The current trusted-kernel execution chain.

1. User RequestLive
Messages enter through HardenedPuppeteer.run().
2. OrchestrationLive
puppeteer.py classifies the task and selects Scout / Forge / Auditor sequence.
3. Trusted RetrievalLive
TrustedMCPAdapter normalizes tool output before models see it.
4. Context AssemblyLive
ContextEngine produces trust-labeled prompt material and basic memory context.
5. Policy-Governed Tool UseLive
ToolPolicy + policy_enforced_call_tool() decide, sanitize, and gate execution.
6. Approval + AuditLive
High-risk paths can be blocked or approval-gated; traces are recorded through checkpoint and observability primitives.

Known Gaps

The project is stronger when these are stated plainly.

No live-service validation path against real local Ollama and Qdrant yet in the automated suite.
Tool coverage is still narrow compared to the long-term vision.
MemRL exists, but is not yet grounded in explicit success/failure outcome labeling.
This is still a trusted-kernel foundation, not a production-autonomous internet agent.

Not Yet Safe To Claim

What future developers should not oversell.

“Fully autonomous” in an unrestricted sense.
“Production secure” for arbitrary external tools or web autonomy.
“Self-improving” in a benchmarked, outcome-validated sense.
“Zero compromise” unless execution scope remains bounded by current policy and testing.

Roadmap

The recommended order for developers joining the project.

Milestone 1: real local-service validation with Ollama and Qdrant under controlled conditions.
Milestone 2: expand tool adapters and validators while keeping all execution policy-centered.
Milestone 3: deepen harness-level tests for richer tool traces, approvals, and cache poisoning fixtures.
Milestone 4: only then widen capability: more tools, richer memory, broader orchestration, measured autonomy.
This brief is meant to be a hiring handoff: it shows what exists, what is safe, what is tested, what is still partial, and what the next engineering move should be.
